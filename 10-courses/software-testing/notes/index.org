#+DATE: <2026-04-22 三>
* 使用说明

本提纲面向只剩一天的冲刺场景设计。
目标是：
1. 在最短时间内吃透高频考点
2. 把常见题型变成固定答题动作
3. 通过模板化训练，确保考试能写出有分答案
* 样题中的考试范围

根据样题与参考答案，本次复习必须覆盖以下五大模块：
1. 测试基础与 V 模型
2. 黑盒测试设计（等价类、边界值、决策表、因果图）
3. 白盒测试（语句覆盖、判定覆盖、条件覆盖、条件组合覆盖、控制流图、环路复杂度、基本路径）
4. 性能测试（响应时间、吞吐量、CPU利用率、瓶颈分析、估算）
5. 测试管理（测试用例、非功能测试、缺陷生命周期、回归测试）


结论：
- 最容易拿分的是：黑盒测试设计、测试管理、基础选择题
- 最容易拉开差距的是：白盒测试中的控制流图、环路复杂度、基本路径
- 性能测试一般体量不大，但容易因为表达空泛或公式不清而丢分
* 总体作战原则
** 原则1：先确保高频模板题，再碰计算题

优先顺序如下：
1. 黑盒测试设计题
2. 测试管理题
3. 基础理论选择题
4. 白盒测试题
5. 性能测试题
** 原则2：每道题都要有固定动作
- 看到“决策表”就先列条件、再列动作、再列规则
- 看到“等价类”就先分有效类与无效类
- 看到“边界值”就先写边界点与边界两侧值
- 看到“控制流图”就找判定节点
- 看到“回归测试”就想到“确认修改+检查是否引入新问题”
** 原则3：所有答案优先写课程口径
- 单元测试、集成测试、系统测试、验收测试
- 黑盒测试、白盒测试
- 语句覆盖、判定覆盖、条件覆盖、条件组合覆盖
- 决策表、等价类、边界值


* 一天冲刺执行计划

** 第一阶段（1小时）：测试基础与V模型

*** 学习目标

- 能区分四个测试级别
- 能说清黑盒测试与白盒测试的区别
- 能写出 V 模型中开发阶段与测试阶段对应的关系
- 能完成这一类的选择题


*** 必须背诵

**** 1. 测试级别
- 单元测试：针对最小可测试单元，重点验证程序设计、模块内部逻辑、接口等
- 集成测试：把模块组装起来后进行测试，重点看模块间接口、调用关系、数据传递
- 系统测试：针对整个系统，验证系统设计目标是否实现，包括功能和非功能特性
- 验收测试：面向用户需求或合同要求，确认系统是否满足用户可接受条件


**** 2. 黑盒与白盒
- 黑盒测试：根据外部功能、输入输出、规格说明来设计测试，不关心内部代码实现
- 白盒测试：根据内部结构、逻辑路径、判定条件来设计测试，关注程序内部执行过程


**** 3. V 模型对照
- 单元测试和集成测试：验证程序设计
- 系统测试：验证系统设计
- 验收测试：追溯用户需求说明


*** 要会的模型动作

**** 看到“V 模型”时
- 需求 \(\leftrightarrow\) 验收测试
- 系统设计 \(\leftrightarrow\) 系统测试
- 详细设计 / 程序设计 \(\leftrightarrow\) 集成测试 / 单元测试


**** 看到“单元测试主要内容”时
优先联想到：
- 模块接口
- 路径
- 边界
- 局部数据结构
- 错误处理


若选项里出现“有效性测试”等偏向系统层面的表述，通常不是单元测试的内容。

*** 本阶段自测：
- [ ] 能不看资料写出四个测试级别
- [ ] 能不看资料画出V模型
- [ ] 能用一句话区分白盒和黑盒测试


** 第二阶段（2小时30分）：黑盒测试设计

*** 学习目标

- 掌握等价类划分法
- 掌握边界值分析法
- 掌握决策表法
- 知道因果图法在课程中的定义，能识别对应题型


*** 本模块的考试定位

这是最稳的得分区，因为步骤清晰、模板统一、评分点明显。

*** 1. 等价类划分法

**** 核心定义
把输入域划分为若干个有代表性的子集合，同一集合中的数据对程序的作用应被认为是“等价”

**** 固定动作

1. 先找输入条件
2. 再分有效等价类和无效等价类
3. 给每个等价类编号
4. 从每个等价类中选一个代表值做测试用例


**** 标准答题模板

#+begin_example
输入条件:______
有效等价类:______
无效等价类:______
测试用例:
1. 输入 ______, 预期 ______
2. 输入 ______, 预期 ______
#+end_example


**** 常见失误
- 只写有效等价类，不写无效类
- 写了有效类，但是没有给出代表测试用例
- 把“合法但不存在的业务值”和“非法输入”混为


*** 2. 边界值分析法

**** 核心定义
错误往往集中出现在输入范围的边界附近，因此重点测试边界及其临界值。

**** 固定动作
若题目给出范围或离散关键点，优先考虑：
- 边界点本身
- 边界内侧值
- 边界外侧值


**** 课程常用写法
- 最大值
- 最大值+1
- 最大值-1
- 最小值
- 最小值+1
- 最小值-1


若是离散合法值，也需要考虑临近值，如 2999/3000/3001

**** 标准答题模板
#+begin_example
边界点: ______
测试值: ______
预期结果: _______
#+end_example

**** 常见失误
- 只测边界点，不测边界点两侧
- 把边界值和等价类混写，答案结构混乱


*** 3. 决策表法
当系统行为由多个条件共同决定，并且不同条件组合会触发不同动作时，优先用决策表

**** 固定动作
1. 列出所有动作
2. 列出每个条件可能取值
3. 列出所有有意义的条件组合
4. 为每种组合写出条件
5. 将每一列规则转成测试用例


**** 决策表的基本结构
- 条件桩：有哪些条件
- 条件项：各一条件在某一规则下取什么值
- 动作桩：系统可能执行的动作
- 动作项：该规则触发哪些动作


**** 决策表转测试用例模板

比如，对于这个例题：


2000年，某高校的标准工资如下：学历为博士，则工资为5000元；学历为硕士，若职称为副教授以上，则工资为5000元，否则工资为4000元；学历为本科，若职称为教授，则工资为5000元，若职称为副教授，则工资为4000元，若职称为讲师及以下，则工资为3000元。


设计决策表为：

| 序号         | 1 | 2 | 3 | 4 | 5 | 6 | 7 |
|--------------+---+---+---+---+---+---+---|
| 条件         |   |   |   |   |   |   |   |
| 学历为博士   | Y | N | N | N | N | N | N |
| 学历为硕士   |   | Y | Y | Y | N | N | N |
| 学历为本科   |   |   |   |   | Y | Y | Y |
| 职称为讲师   |   | Y |   |   | Y |   |   |
| 职称为副教授 |   |   | Y |   |   | Y |   |
| 职称为教授   |   |   |   | Y |   |   | Y |
|--------------+---+---+---+---+---+---+---|
| 动作         |   |   |   |   |   |   |   |
| 工资为5000元 | ✓ |   |   | ✓ |   |   | ✓ |
| 工资为4000元 |   | ✓ | ✓ |   |   | ✓ |   |
| 工资为3000元 |   |   |   |   | ✓ |   |   |


*** 4. 因果图法

**** 课程层面的掌握要求
- 知道它属于黑盒测试用例设计技术
- 知道它属于多个输入条件与输出工作之间逻辑关系复杂的场景
- 若题目给出图或半成品表格，能够按照因果关系补全用例


*** 本阶段训练目标

- [ ] 用工资的例题手写一边决策表
- [ ] 用工资的例题手写一边等价类划分
- [ ] 用工资的例题手写一边边界值测试
- [ ] 口头说出什么时候用决策表，什么时候用等价类

** 第 3 阶段（2 小时 30 分）：白盒测试
*** 学习目标
- 区分语句覆盖、判定覆盖、条件覆盖、条件组合覆盖。
- 能从代码中找判定点。
- 能画出基本控制流图。
- 能计算环路复杂度。
- 能写出基本路径。

*** 3.1 覆盖标准
**** 语句覆盖
让程序中的每条可执行语句至少执行一次。

**** 判定覆盖（分支覆盖）
让每个判定的每个可能结果至少出现一次，即 true 和 false 都要覆盖到。

**** 条件覆盖
让每个判定中的每个基本条件都至少取过一次 true 和 false。

**** 条件组合覆盖 / 多条件覆盖
让判定中各基本条件的所有可能取值组合都至少出现一次。

**** 课程中的强弱关系（冲刺版）
通常可按如下顺序理解：
语句覆盖 < 判定覆盖 < 条件覆盖 / 判定条件覆盖 < 条件组合覆盖

注意：不同教材对“条件判定组合覆盖”“多条件覆盖”“修正条件判定覆盖”的关系表述可能更细。考试时优先按照课堂口径答题。

*** 3.2 固定动作：做覆盖题的步骤
1. 找出代码中的所有判定。
2. 把每个判定拆成逻辑条件。
3. 根据题目要求确定覆盖级别。
4. 写出能满足覆盖要求的输入组合。

**** 看到“判定覆盖”题时怎么做
如果代码里有条件：
#+begin_example
if (A)
#+end_example
那你至少要准备一组让 A 为真，一组让 A 为假。

如果题目像样题那样让你“给出满足 100% DC 所需的逻辑条件”，就直接把每个判定的真假情况列出来。

*** 3.3 控制流图
**** 核心理解
控制流图是把程序执行过程抽象为结点和边的有向图。
- 结点：基本语句块或关键控制点。
- 边：控制流转移关系。

**** 固定动作
1. 把顺序执行的语句合并成块。
2. 遇到 if / while / for 等结构时，单独标出判定结点。
3. 画出 true 分支和 false 分支。
4. 循环结构要画回边。

**** 画图时必须特别注意
- 每个判定节点至少应有两条出边（正常情况下）。
- 循环退出路径不要漏。
- 程序入口与出口要明确。

*** 3.4 环路复杂度 V(G)
**** 最实用的两个公式
#+begin_example
V(G) = E - N + 2
V(G) = 判定节点数 + 1
#+end_example

其中：
- E 是边数。
- N 是结点数。

**** 冲刺建议
考试时优先用“判定节点数 + 1”快速计算；若不放心，再用 E-N+2 复核。

**** 样题口径
你的样题参考答案中，控制流图的环路复杂度给出为 4。做同类题时，要把“路径条数”和“环路复杂度”联系起来记忆：
- 环路复杂度 = 需要考虑的线性独立路径数量。

*** 3.5 基本路径
**** 核心理解
基本路径是指一组线性独立路径，它们组合起来可以覆盖程序的所有边或所有独立控制结构。

**** 固定动作
1. 先算出 V(G)。
2. 再写出与 V(G) 数量一致的独立路径。
3. 每条路径要覆盖一个新的控制流变化。

**** 常见写法
若图中节点编号为 1、2、3……，则路径通常写成：
#+begin_example
1-2-3-5-8
1-2-4-6-8
#+end_example

**** 常见失误
- 路径数目少于 V(G)。
- 写出的多条路径本质重复，没有体现新的独立分支。
- 漏写循环退出路径或循环体路径。

*** 本阶段训练任务
- [ ] 把样题二的答案手抄 1 遍。
- [ ] 自己不看答案重新解释一次：为什么 V(G)=4。
- [ ] 自己用“判定节点数 + 1”公式复算一次。
- [ ] 自己重新写出 4 条基本路径。

*** 本阶段自测
- [ ] 我能区分语句覆盖、判定覆盖、条件覆盖。
- [ ] 我能写出环路复杂度两个公式。
- [ ] 我能说明“为什么路径数和 V(G) 有关系”。

** 第 4 阶段（1 小时 30 分）：性能测试
*** 学习目标
- 会解释响应时间、吞吐量、CPU 利用率。
- 会看表判断瓶颈。
- 会回答性能缺陷成因与解决方案。
- 会做简单估算题。

*** 4.1 必须掌握的术语
- 响应时间：从客户端发起请求到接收到响应结束所经历的时间。
- 吞吐量：单位时间内系统处理的事务数。
- 并发用户数：同一时刻访问系统的用户数。
- CPU 利用率：服务器 CPU 使用情况，过高可能意味着瓶颈。

*** 4.2 看性能表的固定动作
1. 先看并发上升后，吞吐量是否继续提升。
2. 再看响应时间是否显著恶化。
3. 最后看哪一端资源率先接近饱和。

若出现：
- 吞吐量没有明显提升或不满足需求。
- 响应时间明显变长。
- 数据库或应用服务器 CPU 接近打满。
则优先怀疑该侧成为瓶颈。

*** 4.3 样题中的标准分析套路
当并发用户数达到 50 时，数据库 CPU 利用率接近 100%，而应用服务器 CPU 利用率明显较低，因此可以判断数据库端资源使用不合理，数据库很可能成为瓶颈。

**** 常见表述模板
#+begin_example
根据表中数据可知，随着并发用户数上升，数据库服务器 CPU 利用率达到 ________ ，已接近饱和，资源使用不合理；与此同时，交易响应时间增大，说明数据库端可能成为系统瓶颈。
#+end_example

*** 4.4 性能缺陷原因题
可从以下角度回答：
- 数据库资源负载过重。
- 数据库设计或索引设计不合理。
- 单个事务处理时间过长。
- 并发负载增大导致响应时间恶化。

*** 4.5 解决方案题
课程答题时优先写：
- 优化 SQL / 索引 / 表结构。
- 升级服务器资源。
- 做数据库集群或分库分表。
- 优化事务与连接池配置。
- 进行缓存或削峰。

如果题目和样题风格一致，回答“采用数据库集群策略并正确配置”通常有分。

*** 4.6 吞吐量估算题
**** 固定公式思路
#+begin_example
吞吐量 = 总交易量 / 总处理时间
#+end_example

若题目给出 80/20 原理：
- 80% 的业务量集中在 20% 的时间内完成。

则计算步骤通常为：
1. 算出未来总业务量。
2. 取高峰期业务量（如乘 80%）。
3. 算出高峰期有效时间（如乘 20%）。
4. 相除得到高峰吞吐量。

*** 本阶段训练任务

- [ ] 不看解答，自己重写一遍高峰吞吐量算式。
- [ ] 准备 2 句万能性能分析句式。

- [ ] 我看到 CPU 97% 时知道应该怎么判断。

** 第 5 阶段（1 小时 30 分）：测试管理与规范答题
*** 学习目标
- 会写规范测试用例。
- 能列举非功能测试维度。
- 能写缺陷生命周期。
- 能答回归测试与确认测试相关问题。

*** 5.1 测试用例写法
**** 必备结构
- 测试用例编号
- 测试目标
- 测试环境
- 输入数据
- 执行步骤
- 预期结果

**** 万能模板
#+begin_example
测试用例编号：TC-001
测试目标：验证 ________ 功能是否正常。
测试环境：浏览器 / 操作系统 / 网络环境 ________
输入数据：________
执行步骤：
1. ________
2. ________
3. ________
预期结果：________
#+end_example

**** 考场原则
步骤要具体到可操作；预期结果要能验证，不要只写“系统正常”。

*** 5.2 非功能测试维度
课程中常见可答：
- 性能测试
- 压力测试
- 负载测试
- 安全性测试
- 可用性测试
- 可维护性测试
- 可移植性测试
- 兼容性测试

若题目要求“任意 5 个”，优先写自己最熟悉的 5 个。

*** 5.3 缺陷生命周期
课程中常见状态可写为：
- New：新发现。
- Open：已确认并分配处理。
- Fixed：开发已修复，等待验证。
- Closed：验证通过，关闭。

还可补充：
- Rejected / Declined：拒绝修改。
- Deferred：延期处理。

*** 5.4 确认测试与回归测试
**** 确认测试
验证开发修改后的问题是否真的被修复。

**** 回归测试
检查修改是否对原有功能或其他模块造成新的影响。

**** 样题最后一题的评分关键
若开发说修了 7 个问题，测试组不能只拿原来暴露问题的那 5 个测试用例做回归；因为修改可能引入新问题，应该进行更完整的回归测试。

**** 答题模板
#+begin_example
测试组的做法存在问题。
原因是：
1. 对于被认定为“需求问题”的缺陷，不能直接由开发单方面取消，应与用户或项目负责人确认。

* 附：极简背诵版
** 黑盒
- 等价类：分有效、无效；每类取代表值。
- 边界值：取边界点及两侧邻近值。
- 决策表：列条件、列动作、列规则、转用例。

** 白盒
- 语句覆盖：每句执行一次。
- 判定覆盖：每个判定 true/false 都出现。
- 条件覆盖：每个基本条件 true/false 都出现。
- 条件组合覆盖：所有条件组合都出现。
- 环路复杂度：判定节点数 + 1。

** 性能
- 响应时间：请求到响应完成。
- 吞吐量：单位时间处理事务数。
- CPU 接近 100%：高风险瓶颈。

** 管理
- 测试用例：编号、目标、环境、输入、步骤、预期。
- 缺陷流程：New → Open → Fixed → Closed。
- 回归测试：防止改出新问题。
